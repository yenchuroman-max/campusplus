# Контекст проекта для следующей нейросети

Обновлено: 2026-03-05
Проект: **КампусПлюс (СГУГиТ)**

## 1) Что это за система
- Веб-платформа на FastAPI + Jinja2 + SQLite для контроля успеваемости.
- Основные роли: `student`, `teacher`, `admin`.
- Ключевые функции: импорт лекций, генерация тестов, прохождение тестов студентами, аналитика, админ-управление пользователями/группами/дисциплинами.

## 2) Важные текущие договоренности (бизнес-логика)
- Самостоятельная регистрация доступна **только студенту**.
- Преподавателя можно создать только через админские разделы (`/admin/teachers` или `/v1/admin/teachers`).
- Логин в UI называется «логин», но хранится в `users.email` (legacy-совместимость сохранена).
- При ошибке валидации на регистрации форма **не сбрасывается**: ФИО/логин/группа остаются.
- По требованию владельца проекта: данные БД сохранять, не очищать перед тестами/правками.

## 3) Что уже сделано (актуально)

### Backend / auth
- Регистрация студента:
  - `POST /register` принудительно проверяет `role == student`.
  - Валидация логина/пароля/ФИО/группы с возвратом формы и сообщением об ошибке.
- Логин:
  - Вход через поле `login` (с alias `email`), проверка через `validate_login`.
  - Rate-limit по IP (`login_limiter`).
- Админское создание пользователей:
  - Формы создания через `/admin/users/create` и `/v1/admin/users/create` ограничены созданием только роли `teacher`.

### Поиск
- Поиск с хедера работает через `GET /search`.
- На `/` добавлен редирект при `?q=...` на страницу результатов поиска.
- Страница результатов: `app/templates/search_results.html`.

### Главная страница и роли
- Главная (`index.html`) разделена по ролям:
  - Student: личный кабинет, доступные тесты, точки роста.
  - Teacher: панель преподавателя, лекции, аналитика.
  - Admin: студенты, преподаватели, группы/дисциплины.
  - Гость: CTA на вход/регистрацию и ссылки через `next`.
- Приветствие в hero: `Здравствуйте, ФИО`.

### Брендинг/фронт
- Название проекта в интерфейсе и title приведено к: **КампусПлюс**.
- В шапку добавлен знак СГУГиТ без белого фона.
- Подготовлены новые ассеты:
  - `app/static/img/sgugit-logo-clean.png` (прозрачный полный логотип)
  - `app/static/img/sgugit-mark.png` (компактный знак для хедера)
- Тема:
  - По умолчанию при входе — светлая (через `localStorage` ключ `ui-theme`, fallback = `light`).
  - Переключатель темы сделан в стиле проекта (не iOS-клон), с механикой слайдера.
  - Внутри переключателя иконки: луна/солнце по состоянию.
  - Исправлено наложение тумблера на кнопку «Выйти».

## 4) Ключевые файлы (куда смотреть в первую очередь)
- `main.py` — маршруты, авторизация, поиск, регистрация, админские операции.
- `app/templates/base.html` — хедер, навигация, переключатель темы, глобальный поиск.
- `app/templates/index.html` — role-based главная.
- `app/templates/register.html` — форма регистрации студента + сохранение состояния.
- `app/templates/admin_users.html` — создание преподавателей админом.
- `app/static/css/styles.css` — дизайн-система, темы, хедер, тумблер, адаптив.

## 5) Технические заметки
- Проект в текущей папке **не является git-репозиторием** (нет `.git`).
- Для подготовки прозрачного логотипа локально в `.venv` установлен `Pillow`.
- Последняя быстрая проверка после правок: `python3 -m compileall -q app main.py` — проходит.

## 6) Риски/узкие места (что проверить первым делом)
- Визуальные регрессии в хедере на узких ширинах (320–430 px).
- Ролевые переходы и доступы после входа/выхода (student/teacher/admin).
- Поиск по ролям и приватность выдачи (чтобы студент не видел лишнего).
- Генерация тестов и импорт больших PDF (время ответа/таймауты API).

## 7) Мини-сценарий демо перед защитой
1. Войти как админ: показать студентов/преподавателей/группы/дисциплины.
2. Создать преподавателя через админку (логин+пароль).
3. Войти как преподаватель, открыть лекции, показать генерацию теста.
4. Войти как студент, пройти тест, показать результаты и «точки роста».
5. На главной показать переключение темы и ролевой интерфейс.

## 8) Что можно улучшить после защиты (черновой backlog)
- Вынести auth/search/teacher/admin блоки из `main.py` в отдельные роутеры (сейчас файл перегружен).
- Добавить миграции (`alembic`) вместо ручных `ALTER TABLE`.
- Привести модель `email/login` к явным полям в БД (без legacy-путаницы).
- Покрыть критические маршруты интеграционными тестами (регистрация, логин, роли, поиск).
- Добавить кэш/очередь для тяжелой AI-генерации (Celery/RQ + retry + статус задач).
- Ввести централизованный frontend bundle (минимизировать inline JS в шаблонах).
- Сделать отдельный режим "демо" с предзаполненным seed и reset-сценарием для комиссии.
